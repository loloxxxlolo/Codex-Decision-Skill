# Codex Decision Skill

一个基于 Codex 通过记录文本去实现辅助决策和回想的 skill。

我做这个 skill 的初衷很简单：我是一个很容易反复内耗、反复纠结的人。很多问题其实之前已经想清楚了，也已经得出过结论，但过一段时间后又会忘记，然后重新问、重新想、重新纠结一遍。

所以我希望 AI 能帮我把之前已经讨论清楚的内容记录下来。等我下次又问到类似问题时，它可以直接参考过去的结论，提醒我当时是怎么判断的，以及现在应该怎么做。

后来我发现，它不只是能辅助决策。只要长期记录的背景足够多，它也可以帮我做计划、回顾项目、整理下一步任务，减少很多“重新回想”的成本。

尤其是对我这种时间一久就容易忘、回顾起来又很痛苦的人来说，这个 skill 确实能帮我省掉不少不必要的内耗。



## 解决什么问题

- 方向选择反复纠结。
- 项目是否继续、暂停、放弃没有标准。
- 上次已经聊过的结论，下次又被当成新问题。
- AI 只回答当前问题，不承接历史上下文。
- 决策讨论结束后，没有稳定写回记忆。

## 不解决什么问题

- 不做全量人生记忆。
- 不存所有聊天全文。
- 不用于普通技术排错。
- 不用于一次性事实查询。
- 不替用户脑补计划、风险或待办。
- 不追求复杂知识库或云端同步。

## 核心流程

1. 判断问题是否属于历史决策主题。
2. 先读取短期记忆。
3. 只读取相关长期项目记忆。
4. 先回顾已有结论，再回答当前问题。
5. 如果形成稳定结论，再压缩写回记忆。


如果你不想使用中文目录，也可以改成：

```text
memory/
  short-term-memory.md
  long-term-memory/
    decision-board.md
    project-index.md
    projects/
```

## 关键原则

- 只记录稳定结论，不记录每次情绪波动。
- AI 的建议只有在用户确认后才写入长期记忆。
- 每个项目最好有判断标准：继续、暂缓、升级、放弃。
- 当前决策看板只保留当前最重要的几件事。
- 不要为了模板完整而扩写低价值内容。

## 建议仓库名

```text
codex-decision-skill
```

## 文件说明

- `SKILL.md`：Codex skill 主文件。
- `SKILL.zh-CN.md`：中文精简版 skill 文件。
- `README_EN.md`：英文说明。
- `templates/`：短期记忆、长期项目、当前决策看板模板。
- `examples/`：旅行决策、项目取舍、记忆写回案例。

## 使用方式

## 快速开始

### 1. 安装 skill

如果你主要用英文或想用更通用版本，复制 `SKILL.md` 到你的 Codex skill 目录。

如果你主要用中文，建议复制 `SKILL.zh-CN.md`，并重命名为 `SKILL.md`：

```text
~/.codex/skills/decision-skill/SKILL.md
```

Windows 示例：

```text
C:\Users\<your-name>\.codex\skills\decision-skill\SKILL.md
```

### 2. 在你的项目工作区创建记忆目录

复制 `templates/` 里的模板，得到类似结构：

```text
记忆/
  短期记忆.md
  长期记忆/
    当前决策看板.md
    项目列表/
      your_project.md
```

如果你更喜欢英文目录，也可以用：

```text
memory/
  short-term-memory.md
  long-term-memory/
    decision-board.md
    projects/
      your-project.md
```

### 3. 开始使用

在 Codex 里提问时，可以直接说：

```text
@decision-skill，帮我判断这个项目要不要继续。
```

或：

```text
@decision-skill，把这次结论写入长期记忆。
```

### 4. 推荐使用方式

- 先维护 `短期记忆.md`，保存最近关注点。
- 再维护 `当前决策看板.md`，保存当前最重要的 1~3 件事。
- 每个长期项目单独一个 `.md` 文件。
- 每次只更新稳定结论，不保存完整聊天记录。

### 示例 1：是否公开发布一个 Codex skill

用户问题：

```text
我做了一个 Codex skill，要不要发到 GitHub？会不会变成额外负担？
```

skill 应该先恢复已有上下文：

- 当前阶段不适合把新项目做成高压任务。
- 这个 skill 已经能形成最小闭环。
- 发布的主要意义是整理、展示和复用，不是马上获取流量。

输出判断：

```text
可以发布，但只做 v0.1 最小公开版。
范围限定为 README、SKILL 文件、模板和少量案例。
不要做官网、视频教程、复杂安装脚本或高压运营。
```

可写入记忆：

```md
- decision skill 可以发布到 GitHub，但只做 v0.1 最小公开版。
- 发布目标是整理、展示和复用，不是正式运营开源项目。
```

### 示例 2：移动记录 App 要不要加总结功能

用户问题：

```text
我的记录 App 已经能手动记录生活了，下一步要不要加总结功能？
```

skill 应该恢复已有上下文：

- 记录类产品的核心是快速记录。
- App 已经有基础记录闭环。
- 当前不适合过早做复杂云端系统。

输出判断：

```text
可以加总结功能，但优先做温柔生活总结和周/月回顾。
不要一上来做复杂报告、年度总结或云端汇总。
```

可写入判断标准：

```md
优先升级：
- 能把已记录内容转成温柔生活总结。
- 能提升回看价值，例如周回顾、月回顾、收藏记录。

暂缓：
- 总结功能会拖慢记录速度。
- 还没证明本地记录闭环稳定，就提前做复杂 AI 汇总。
```

### 示例 3：内容项目要不要投放推广

用户问题：

```text
一个垂直内容账号刚开始做，要不要投放推广？投多少？怎么看有没有价值？
```

skill 应该恢复已有上下文：

- 账号还在早期验证阶段。
- 不适合一开始就把自己放到投资人或老板位置。
- 需要先验证内容方向和协作稳定性。

输出判断：

```text
可以小额测试，但投放目标不是买曝光，而是验证选题和真实需求。
先看收藏、评论、私信、转化线索，而不是只看播放量。
```

可写入判断标准：

```md
继续推进：
- 小额投放后出现真实需求信号。
- 能找到 1~2 个更值得持续测试的选题。

不升级投入：
- 没有验证出稳定选题前，不加大预算。
- 没有验证协作稳定前，不扩大投入角色。
```

### 示例 4：长期旅行和生活节奏怎么排

用户问题：

```text
我接下来几个月是继续工作攒缓冲，还是直接开始长线旅行？
```

skill 应该恢复已有上下文：

- 当前阶段主线是现金缓冲和状态修复。
- 长线旅行不应该被当成逃避现实问题的答案。
- 可以先安排短线旅行作为过渡。

输出判断：

```text
先保留工作和状态修复主线。
短线旅行可以做，长线旅行等现金、状态和交接都可控后再启动。
```

可写入判断标准：

```md
继续长线旅行：
- 现金、生活状态和交接准备都可控。

降级为短线或分段：
- 状态一般，但仍想保留旅行体验。

暂缓旅行：
- 预算不足，或旅行正在变成逃避现实问题。
```

## v0.1 边界

这个版本只追求一个最小闭环：

> 同一个问题第二次出现时，不要像第一次见到一样回答。

后续再考虑自动化、同步、图形界面或云端汇总。
